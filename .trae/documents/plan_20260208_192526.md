O sistema será desenvolvido como um módulo backend integrado ao servidor Node.js existente, utilizando Firebase Firestore para persistência e filas.

### 1. Arquitetura do Backend (Server)
O sistema será composto por três serviços principais e uma camada de API:

*   **NotificationService**: Responsável pela lógica de negócio.
    *   *Scheduler Diário*: Verifica assinaturas vencendo em 30, 15, 7 e 1 dia.
    *   *Generator*: Cria jobs de notificação com base nas preferências do usuário e templates ativos.
*   **QueueSystem (Firestore-backed)**:
    *   Implementação de uma fila robusta utilizando a coleção `notification_queue` no Firestore.
    *   Suporte a estados: `pending`, `processing`, `completed`, `failed`.
    *   Worker process que roda a cada minuto para processar a fila, com retry logic (tentativas de reenvio) e fallback de canais (Push -> Email -> SMS).
*   **ChannelService**:
    *   **Email**: Integração com `nodemailer` (suporte a SMTP/Gmail).
    *   **Push**: Integração com `expo-server-sdk` para envio aos dispositivos móveis.
    *   **SMS**: Interface preparada para integração (mock inicial, pronto para Twilio/SNS).
*   **TemplateEngine**:
    *   Uso de `handlebars` para renderização dinâmica de mensagens (ex: "Olá {{name}}, sua assinatura do {{service}} vence em {{days}} dias").

### 2. Banco de Dados (Firestore)
Novas coleções/estruturas:
*   `notification_preferences`: Configurações do usuário (canais ativos, horários de silêncio).
*   `notification_templates`: Modelos de mensagens editáveis via API.
*   `notification_history`: Log auditável de todos os envios.
*   `notification_queue`: Fila de processamento.

### 3. API RESTful
Novos endpoints em `server/api/notifications.js`:
*   `GET/PUT /api/notifications/preferences`: Gerenciamento de preferências do usuário.
*   `GET /api/notifications/history`: Histórico de notificações para o usuário.
*   `POST /api/notifications/templates`: (Admin) Criação/Edição de templates.
*   `GET /api/notifications/stats`: (Admin) Dashboard com taxas de envio e falhas.

### 4. Frontend (App)
*   Criação da tela `app/settings/notifications.tsx`:
    *   Interface para ativar/desativar canais (Email, Push, SMS).
    *   Configuração de antecedência de alertas.

### 5. Plano de Implementação
1.  **Instalação de Dependências**: `node-cron`, `nodemailer`, `expo-server-sdk`, `handlebars`.
2.  **Implementação dos Serviços**: Criar a estrutura de pastas `server/services` e implementar a lógica de agendamento e envio.
3.  **Configuração da API**: Criar as rotas e integrar ao `server/index.js`.
4.  **Frontend**: Criar a tela de configurações no app.
5.  **Testes**: Script de teste de integração para validar o fluxo completo (Criação de Assinatura -> Agendamento -> Fila -> Envio).
